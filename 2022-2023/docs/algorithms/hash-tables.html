Hash Tables

Written By Ved Panse Last Modified: 30 June 2021

In this section of the documentation, we will be learning about the implementation, collisions, and functions of hash tables.

What is a hash table?

Hash table is one of the most useful and widely used data structure in the world.

Imagine yourself to be a teacher who wants to select random students for a school play, but there should be a maximum of 5 students from each class (out of 18 classes). The list given to you is unsorted. In such a case, it will take you a lot of time to find the name of students.

This process would take O(n) time.

On the other hand, if you are given a sorted list of students, you will be able to find the student's name along with his/her class section without taking much time. Thus, this scenario would take O(log n) time.

Even with a sorted list of students, you will take a considerable amount of time. You will need a person who can memorize the location of each student's name and navigate you accordingly. How can you find such a person? This is where hash functions come in.



Hash Functions

A hash function is a function in which you input a string and get back a number.

A hash function matches a string to a number. You may think that there are no desirable patterns of numbers that get associated with a string. However, there are two conditions.

  1. Suppose you input John, and if you get "98" back, then each time you input John, you should get "98" back (since marks do not change).
  2. It should map different words to different numbers. This means that it should not assign a single number, let's say "30" to all the strings (since it is very unlikely that all the students score the same marks).



How is this related to our topic?

It turns out that the above algorithm is quite helpful to make the "person" that we talked about. Let us name that person "Clay" so that we can refer to the example easily in the future.



Making Clay

Here's the algorithm by which we will be able to make Clay. Input a random name, say George into the hash function. Let's say that the function attributes "George" to "10". We will then create an array and store our names and grade in that array. Then, we will insert "George" along with his experience in school plays at index 10.

Continuing this algorithm, we will end up with an array full of students. Now, if you want to find the class section of a random student, say "Aleesha". You will no longer have to look into the array or the unsorted list. You can directly feed "Aleesha" to the hash function and get its position in the array (recall condition 1) to get the experience. You have now created your clay.

The hash function tells you directly where an item is stored.



The Hash Table

Combining a hash function and an array, you will get a hash table. It is way different from arrays and lists, which store data directly in the computer's memory. Hash tables, on the other hand, are smarter, and a lot faster.

If you are experienced in Python, you might have already learned hash tables. They are called dictionaries in Python.

Caching

Caching can be considered as an example of hash tables too. Caching is just as simple as making the client-side devices (customer's devices) memorize the data that they search over and over again. For example, if you search "Java" over and over again in Google, you may get either of the results:

  • Java: A programming language
  • Java: A country in South-East Asia

The google website saves your behavior in the form of cache on your device to match the results to your query. This also helps to reduce the load on Google's server, making the process a whole lot faster in the long run.



How is this relevant to our topic?

Caching is another example of a hash table since the data stored as cache is stored in the form of a hash. A key, maybe the query you searched for, is mapped to the number of times of which you searched for that query.

Google then shows you data relevant to the "most-searched" data by you.

Collisions

Collisions occur when two keys get assigned the same slot in the array. The name "Adrienne" in our hash table may overwrite "Adrian" in the array. A common solution to prevent such collisions is by using linked lists in that hash table. But, this would slow down its speed.



Performance of Hash Tables

Hash tables take O(1) time, which means that it is constant. Thus, its performance doesn't depend on how large a hash table is. A hash table with 1 billion elements will produce output just as fast as another hash table with 1 element.



Comparison of Hash Tables with Arrays and Lists

By looking at the average case of hash tables, they are as fast as arrays at searching. They are also as fast as linked lists when it comes to deletions and insertions. The average case scenario thus seems to be the best in both contexts. However, when we look at the worst-case scenario, it is the worst of both worlds.

Worst-case scenarios mainly involve collisions. Thus, avoiding collisions will simply prevent the worst-case scenario from occurring.
For this, you may need:

  • A low load factor...(see below for definition)
  • A good hash function...(having values arranged orderly)



Load Factor

The load factor can be calculated by dividing the number of items in a hash table by the total number of slots. Thus, it calculates the number of slots occupied in a given array.

The best-case scenario is when the load factor is less than 1. It would imply that each item in the array has at least 1 slot to itself. The worst-case scenario is when the load factor is more than 1. It would imply that not all items in an array have an entire slot to themselves.

To avoid the worst-case scenario that was mentioned above, we will need to resize the hash table to probably twice the size.

It is generally recommended to resize the array when the load factor grows equal to or more than 0.7. Resizing is expensive and may cost a lot of time (can be subtly experienced by the user). But it turns out that hash tables take O(1) time even including resizing.
