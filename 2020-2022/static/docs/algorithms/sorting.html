Sorting Algorithms

Written By Ved Panse Last Modified: 30 June 2021

In this section, we will discuss different types of sorting algorithms. Even a small topic, sorting, has a great effort put in it.

What are sorting algorithms?

Suppose you are given a list and an array, each containing billions of elements. You are assigned to sort all the elements in alphabetical order. To maximize your program, you use an algorithm that gets you the output in the minimum time.



What are different types of sorting algorithms?

In this documentation, we will be broadly classifying the sorting algorithms into two parts:

  1. Selection Sort
  2. Quick Sort



Introduction to Selection Sort

This algorithm is neat-looking, but not necessarily fast. In this documentation, we will be focussing on Quick Sort algorithms.



Introduction to Quick Sort

Quicksort is a well-known recursive technique used to solve problems. It is a type of Divide-and-Conquer Technique (D&C). It is a sorting algorithm and a much faster one as compared to selection sort. It can be solved in the following two ways:

  1. Figure out the base case. It should be the simplest case.
  2. Divide or decrease your problem until it becomes the base case

Merge Sort

This is another sorting algorithm, with a big O notation of O(n log n). Much faster! Quicksort is a tricky case. If we consider the worst-case scenario, quicksort would take O(n^2) time. Thus, this case of quicksort is as slow as the selection sort.



Merge Sort vs Quicksort

You might be wondering what the difference between the speeds of merge sort and quick sort is. Well, the big O notation has a constant (let's say c) for each algorithm case.



This constant makes a difference here. Quicksort has a smaller constant than merge sort. Being inversely proportional, if they are both O(n log n) time, quicksort is faster. Moreover, the quicksort hits the average case (the opposite of worst-case) way more often than the merge case. Hence, quicksort is faster and better.

Note: The constant never matters when comparing simple search and binary search since the difference between O(log n) and O(n) is too large to be filled.

Average Case VS Worst Case

You might also be wondering what the average cases and worst cases are. This would require us to ponder over back to our sorting algorithm where we had chosen a pivot (to be the first element of the array).

Case 1 Case 2
If we always choose the first element of the pivot, the two subarrays do not get divided equally. Moreover, one of the subarrays always remains empty. The other subarray becomes too long! However, if we chose a middle element to be the pivot, both arrays would have almost an equal distribution of elements. It would make the subarrays shorter considerably.


The first case that we saw is an example of the worst-case scenario since we touch all the elements in the array. Thus, we touch the O(n) functions of the call stack. The second case, on the other hand, is an example of the best-case scenario, since we take O (log n) time as the height of the call stack.
